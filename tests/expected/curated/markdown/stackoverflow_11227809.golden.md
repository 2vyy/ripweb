Why is processing a sorted array faster than processing an unsorted array? The code demonstrates this with a C++ benchmark: without sorting, 11.54 seconds; with sorted data, 1.93 seconds (~6x faster). This is about branch prediction, not caching. You are a victim of branch prediction fail. Consider a railroad junction: a blind operator must stop the train to ask which way to go. Better to guess! If you guess right, the train continues; if wrong, it stops, backs up, and you flip the switch. Modern processors have long pipelines, so misprediction is very expensive. The culprit is the if-statement: if (data[c] >= 128) sum += data[c]; With data evenly distributed between 0-255, sorted data is highly predictable (first half = false, second half = true), while unsorted data is random and impossible to predict. Core i7 920 benchmarks show: Branching - Random data: 11.777s, Branching - Sorted data: 2.352s, Branchless - Random data: 2.564s, Branchless - Sorted data: 2.587s. With the branchless hack, there's no difference between sorted/unsorted. The sorted case is branch-prediction friendly: data = 0, 1, 2, ... 126, 127, 128, 129, ... 250, 251, 252; branch = NNNNNNNNNNNN ... NNNNNNNTTTTTTTTT ... TTTTTTTTTT (easy!). The unsorted case is random: data = 226, 185, 125, 158, 198, 144, 217, 79, 202, 118...; branch = TTNTTTTNTNNTTT ... (completely random - impossible!). In this case, the CPU is waiting for the branch to resolve. It doesn't know which way to go, so it can't fill the pipeline with instructions. The CPU will sit idle. How much? If 50% of the branches are taken, the pipeline must be flushed for half of them. In this case, the branch is data[c] >= 128. Half the time, data[c] is < 128 (branch not taken), and half the time it's >= 128 (branch taken). That's a 50% misprediction rate! With sorted data, the first 128 iterations have data < 128 (branch not taken), and the next 128 have data >= 128 (branch taken). The branch predictor will correctly predict the branch every time after the first few iterations. With unsorted data, the branch predictor has no idea which way the branch will go. It will be wrong 50% of the time. In the benchmark, sorting the data allows the branch predictor to correctly predict the branch. Without sorting, the branch predictor is wrong 50% of the time, causing the pipeline to be flushed frequently, which slows down the execution. The branchless version uses a trick: sum += data[c] & - (data[c] >= 128); This bitwise trick avoids the branch. Instead of using an if-statement, it uses a mask to conditionally add the value. The - (data[c] >= 128) will be -1 when the condition is true (>= 128), and 0 when it's false. This allows the sum to be updated without a branch. With this trick, there's no branch to predict, so the performance is the same for sorted and unsorted data.
